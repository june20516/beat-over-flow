# BeatOverflow 설계 문서

> 작성일: 2026-05-26
> 상태: 설계 합의 완료 (구현 계획 작성 전)

## 1. 개요

BeatOverflow는 **베이스 플로우(기준이 되는 음악) 위에 사운드 트랙을 쌓아 비트/편곡을 만들고, 그것을 리듬게임처럼 연주하는 클라이언트 전용 웹 게임**이다.

핵심 경험은 **"만들기(작곡) ↔ 연주하기(게임)" 순환** 그 자체이며, 두 경험을 동등한 1급 시민으로 다룬다.

대상 사용자에는 **음악 전문성이 없는 캐주얼 유저**가 포함된다. DAW에 익숙하지 않아도 쓸 수 있도록 접근성을 우선한다.

서버를 두지 않는다. 모든 동작은 브라우저 내에서 완결되며, 데이터는 로컬(IndexedDB)에 저장된다.

## 2. 용어 정의

- **베이스 플로우 (Base Flow)**: 기준이 되는 음악. v1에서는 업로드한 오디오 파일. 모든 모드에서 항상 재생되는 마스터 클럭이자 반주.
- **트랙 (Track)**: 베이스 플로우와 평행하게 표현되는 레인. 하나의 사운드 리소스를 할당받는다.
- **마커 (Marker)**: 트랙 위 특정 시각에 사운드를 재생하라는 표시. 연속 타임스탬프(ms)로 저장.
- **사운드 리소스 (Sound)**: 트랙이 재생하는 샘플. 내장 CC0 라이브러리 또는 유저 업로드.
- **전역 모드 (Global Mode)**: 지금 사용자가 하는 활동 — 리스닝 / 플레이 / 레코드.
- **트랙 상태 (Track Status)**: 각 트랙의 역할 — Mute / Listening / Play / Write.

## 3. 핵심 메커니즘 — 전역 모드 × 트랙 상태 (곱셈 모델)

> 이 게임의 심장. 전역 모드(활동)와 트랙 상태(역할)가 **곱해져** 동작이 결정된다.

**핵심 규칙**: 트랙은 자기 상태가 현재 모드와 "교차"할 때만 인터랙티브해지고, 그 외에는 기존 마커를 자동 재생한다. **Mute는 항상 무음. 베이스 플로우는 모든 모드에서 항상 재생.**

| 트랙 상태 ↓ \ 전역 모드 → | 🎧 리스닝 | 🎮 플레이 | ⏺ 레코드 |
|---|---|---|---|
| **Mute** | 무음 | 무음 | 무음 |
| **Listening** (수동) | 자동재생 | 자동재생(반주) | 자동재생(참고) |
| **Play** (연주용) | 자동재생 | **키 입력 요구 + 채점** | 자동재생(참고) |
| **Write** (기록용) | 자동재생 | 자동재생 | **키/클릭으로 마커 기록** |

즉 트랙 상태는 "이 트랙이 **어느 모드에서 인터랙티브해질지**"를 선언한다:
- **Mute** → 절대 안 울림
- **Listening** → 항상 수동(자동재생만)
- **Play** → 플레이 모드에서만 사용자가 연주(그 외엔 자동재생)
- **Write** → 레코드 모드에서만 사용자가 기록(그 외엔 자동재생)

세부 전제:
1. **레코드 모드의 Write 트랙은 오버더빙**: 기존 마커는 들리면서 새 키 입력이 마커로 추가된다. (덮어쓰기 모드는 추후)
2. **플레이 모드에서 여러 Play 트랙이 동시에** 각자의 키로 채점될 수 있다(멀티 레인).
3. 마커는 **연속 타임스탬프(ms)**로 저장한다. 스텝 시퀀서 그리드는 편집 보조일 뿐, 데이터는 자유 타이밍.

## 4. 데이터 모델 (TypeScript)

```ts
type TrackStatus = "mute" | "listening" | "play" | "write";
type GlobalMode  = "listening" | "play" | "record";

interface Project {
  id: string;
  name: string;
  createdAt: number;
  updatedAt: number;
  baseFlow: BaseFlowRef;
  tracks: Track[];
  master: { volume: number };
}

// 베이스 플로우는 추상화된 참조. v1은 audioFile만, youtube는 v2에서 분기 추가.
type BaseFlowRef =
  | { kind: "audioFile"; assetId: string; durationMs: number }
  | { kind: "youtube"; videoId: string; durationMs?: number }; // v2

interface Track {
  id: string;
  name: string;
  status: TrackStatus;
  sound: SoundRef;
  keyBinding: string | null;   // KeyboardEvent.code, 예: "KeyA"
  markers: Marker[];
  volume: number;
  color: string;
}

type SoundRef =
  | { kind: "builtin"; sampleId: string }  // 내장 CC0 라이브러리
  | { kind: "upload";  assetId: string };  // 유저 업로드

interface Marker {
  id: string;
  timeMs: number;   // 베이스 플로우 시작 기준 연속 타임스탬프
}
```

오디오 바이너리(베이스 플로우 파일 + 업로드 샘플)는 `Asset { id, blob, name }`로 **IndexedDB에 별도 저장**하고 `assetId`로 참조한다. 프로젝트 메타(JSON)는 변경 시 디바운스 자동저장.

## 5. 아키텍처 — 6개 레이어

각 레이어는 단일 책임을 가지며 독립적으로 이해·테스트 가능해야 한다.
**핵심 원칙: 오디오 엔진은 React를 모르고, 스토어가 단일 진실원천이며, Canvas는 클럭+스토어만 읽고, 입력은 스토어/스코어링으로 흐른다.**

1. **오디오 엔진 (audio)**
   - `AudioContext` 래퍼, 마스터 출력, 디코딩.
   - `BaseFlowSource` 인터페이스: `play() / pause() / seek(ms) / currentTimeMs / durationMs`. v1 구현체 `AudioFileSource`(디코드된 `AudioBuffer` 재생). 유튜브는 v2에서 `YouTubeSource`로 동일 인터페이스 구현.
   - `Scheduler` — **lookahead 스케줄링**("A Tale of Two Clocks" 패턴): `setInterval` 룩어헤드 + `audioCtx.currentTime` 기반 정밀 예약. 다가오는 자동재생 마커를 미리 예약한다.
   - `SampleLibrary` — 내장 CC0 샘플 레지스트리 + 로더, 업로드 샘플 디코딩 캐시.
2. **클럭 (clock)**
   - 베이스 플로우 재생 위치가 마스터 클럭. 마커시각 → `ctxStartTime + (markerMs − startOffsetMs)`로 매핑.
   - 채점용 입력 시각은 동일한 `audioCtx` 시간축에서 캡처해 가장 가까운 마커와 비교.
3. **상태 (store, Zustand)**
   - Project·tracks·markers·전역모드·트랜스포트·플레이헤드·선택 상태. 단일 진실원천.
   - 모드×상태 동작은 순수함수로 도출(`resolveTrackBehavior(mode, status)`).
4. **입력 (input)**
   - `KeyboardController` — `keydown`을 키 바인딩으로 트랙에 매핑. 모드에 따라 분기:
     - 플레이 모드 → `ScoringEngine`(판정/점수/콤보)
     - 레코드 모드 → 현재 플레이헤드 시각에 마커 추가(+소리 피드백)
   - `StepSequencer` — 구간 + 칸수 → 마커 생성/반복 채우기(§6).
5. **렌더링 (render)**
   - `TimelineCanvas` — Canvas 60fps(`requestAnimationFrame`): 베이스 파형 레인, 트랙 레인, 마커, 플레이헤드, 그리드 오버레이, 히트 피드백. 클럭+스토어만 읽음.
   - React UI 셸 — 트랜스포트 바, 모드 스위처, 트랙 헤더, 프로젝트 목록, 스텝 시퀀서 패널, 점수 HUD.
6. **지속성 (persistence)**
   - `ProjectRepository` — IndexedDB(`idb` 라이브러리). 프로젝트 메타 + 에셋 CRUD, 디바운스 자동저장.
   - `ExportService` — 추후(공유 기능).

## 6. 타이밍 그리드 & 스텝 시퀀서

베이스 플로우는 BPM을 모르는 임의 오디오다. v1은 **구간 균등분할** 방식을 기본으로 한다(BPM/박자 그리드는 v2 확장).

- **구간(Region)**: 시퀀서의 이동·크기조절 가능한 편집 창. 타임라인 위에서 드래그로 옮기고 양끝을 늘렸다 줄인다.
- **칸(Steps)**: 현재 구간을 N칸으로 균등분할. 각 칸 시각 = `regionStart + i*(regionEnd-regionStart)/N`.
- **패턴(Pattern)**: 켜진 칸 집합. 칸을 켜면 그 시각에 마커 생성, 끄면 그 마커 삭제.
- **읽기(Read-back)**: 시퀀서는 현재 그리드 칸에 (오차 허용 범위 내로) 정렬된 마커를 "켜짐"으로 표시. 그리드에 안 맞는 자유 마커는 타임라인에는 보이되 칸으로는 표시 안 함.

### 반복 채우기 (Repeat Fill)

찍기(Stamp) 모델: 반복 채우기는 그 순간 패턴을 **실제 마커로 복제해 박는다.** 영속적 패턴 객체는 없다.

- 대상 범위(target) 선택: **N회 반복** / **지정 지점까지** / **곡 끝까지**.
  - 예: 15초 구간 패턴 + "1:30까지" → 6번 타일링되어 0:00~1:30에 마커가 박힘.
- 이후 구간을 다른 곳으로 옮기고 칸수·패턴을 새로 짜서 다른 범위를 채울 수 있다. 서로 다른 패턴의 섹션이 한 곡 안에 공존하며, 전부 단순 마커라 충돌이 없다. 자유 마커(마우스·키)도 사이에 섞여 공존.
- **B(라이브 루프) 대체 오버라이드**: "이 범위 지우고 다시 채우기" 한 단계로 패턴 수정 후 재적용 = 라이브 수정과 동일 결과. 영속 객체 없이 충족.

## 7. 마커 입력 방식 (v1 모두 포함)

세 방식 모두 동일한 단일 마커를 생성한다:
1. **스텝 시퀀서** — §6.
2. **레인 마우스 클릭** — 레코드 모드에서 트랙 레인의 특정 위치를 클릭해 그 시각에 마커 배치.
3. **레코드 중 키 입력** — 레코드 모드에서 Write 트랙의 매핑된 키를 누르면 현재 플레이헤드 시각에 마커 기록(오버더빙).

추후(v2+): 일정 간격 자동 생성, 숫자 타임스탬프 직접 입력.

## 8. 플레이 모드 채점

리듬게임다운 정확도 채점. 캐주얼 친화로 너그럽게, 추후 조정 가능.

- 판정 윈도우: **Perfect ±40ms / Good ±100ms / Miss** (그 밖 또는 입력 없음).
- 점수: Perfect 100, Good 50, Miss 0.
- 콤보: Perfect/Good에 증가, Miss에 리셋.
- 정확도% = 획득 점수 / 총합 점수.
- 마커 없는 곳의 **여분 키 입력**: 소리는 나되 감점 없음.
- 여러 Play 트랙의 점수는 **합산**.

## 9. 저장 / 지속성

- **IndexedDB 자동저장 + 다중 프로젝트**.
- 작업이 자동 저장되어 새로고침에도 유지. 여러 곡을 명명·관리하는 프로젝트 목록 제공.
- 에셋(오디오 blob)과 프로젝트 메타(JSON)를 분리 저장, `assetId`로 참조.
- 내보내기/공유는 추후(§13).

## 10. 화면 구조 / UX

- **프로젝트 목록(홈)**: 새 프로젝트 / 열기 / 이름변경 / 삭제.
- **에디터(메인, 단일 화면)**:
  - 상단: 트랜스포트(재생/정지, 시각, **전역 모드 스위처**, 마스터 볼륨).
  - 중앙: `TimelineCanvas` — 베이스 레인(위) + 트랙 레인(아래). 시간은 좌→우, 플레이헤드 수직선이 좌→우 스윕.
  - 각 레인 좌측: 트랙 헤더(이름·상태·키 바인딩·사운드 선택·볼륨·뮤트).
  - 하단 패널: 선택된 트랙의 스텝 시퀀서 + 구간/반복 설정.
  - 플레이 모드: 점수 HUD 오버레이.
- **첫 실행**: 베이스 플로우 업로드 유도.
- **레이아웃 방향**: v1은 **가로 타임라인(DAW 스타일)**. 세로 낙하 컬럼(리듬게임 스타일)은 **플레이 모드 전용 뷰로 v2에서 전환 제공**.

## 11. 기술 스택

- **React + Vite + TypeScript** (UI 셸).
- **Canvas** — 타임라인/플레이헤드 60fps 렌더링.
- **Raw Web Audio API + 커스텀 lookahead 스케줄러** — 정밀 타이밍. (Tone.js 미사용)
- **Zustand** — 상태 관리.
- **idb** — IndexedDB 래퍼.
- TypeScript에서 `any` 지양, 가독성 우선.

## 12. v1 범위

### 포함 (IN)
- 베이스 플로우 오디오 파일 업로드 → 디코딩 → 파형 표시
- 내장 CC0 샘플 라이브러리 + 샘플 업로드
- 트랙 생성/사운드 할당/키 바인딩/상태/볼륨/색상
- 가로 타임라인(A안): 베이스 레인 + 트랙 레인, 마커, 플레이헤드, 구간 균등분할 그리드
- 스텝 시퀀서 패널(구간 + 칸수 + 반복 채우기)
- 마커 입력 3종(스텝 시퀀서 · 마우스 클릭 · 키 녹음)
- 전역 모드 3종(리스닝/플레이/레코드) × 트랙 상태 4종
- 플레이 채점(Perfect/Good/Miss, 점수·콤보·정확도%)
- IndexedDB 자동저장 + 다중 프로젝트

### 제외 (추후)
유튜브 소스 + 세로 낙하 플레이뷰(B안) · BPM/박자 그리드 · 인터벌 자동생성 & 숫자 타임스탬프 입력 · 내보내기/공유 · 벨로시티/이펙트/마커 속성 · 덮어쓰기 녹음

## 13. 향후 확장 (v2+)

- **YouTube 베이스 플로우**: `YouTubeSource`를 `BaseFlowSource`로 추가. iframe Player API 기반. 타이밍은 currentTime 폴링(~250ms)으로 **느슨함** — 정밀 채점보다 리스닝/캐주얼 연주 중심. (서버 추출은 ToS 위반이라 배제)
- **세로 낙하 플레이뷰(B안)**: 플레이 모드에서 전환 가능한 리듬게임 스타일 뷰.
- **BPM/박자 그리드**: 파워유저용. 탭템포 BPM 추정 보조.
- **마커 입력 추가**: 일정 간격 자동 생성, 숫자 타임스탬프 입력.
- **공유**: 작품(마커 + 트랙 설정 + 베이스 참조)을 압축 문자열/쿼리파람/링크 또는 파일 포맷으로 내보내기·불러오기. 업로드 음원 포함 시 공유 책임은 공유자에게.

## 14. 열린 세부사항 (구현 중 결정)

- 시퀀서 그리드 read-back의 마커 정렬 오차 허용 범위(tolerance) 구체값.
- 트랙 수 상한, 샘플 동시 재생(폴리포니) 상한.
- 베이스 플로우 파형 피크 계산 방식(다운샘플 해상도).
